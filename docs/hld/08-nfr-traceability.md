# 08-nfr-traceability.md — NFR Traceability Matrix

This document maps all 42 Non-Functional Requirements (NFRs) from Section 5 of the SRS to their respective Architectural Decision Records (ADRs), explicit non-architectural rationale, or approved out-of-scope decisions.

## NFR Traceability Matrix

Rows marked `_Non-Architectural_` are requirements with no LLD-level design decision. Each such row must name the owning non-design artifact in the residual-risk column. Requirements with runtime, extension, security, filesystem, packaging, or process-behavior impact must map to an ADR.

## HLD-Level Follow-Up for LLD

The following NFRs have HLD architectural ownership and require detailed budgets or interfaces during LLD:

1. **NFR-PERF-001 (Log Rendering Latency < 100 ms)** — `ADR-005` bounds log ingestion and memory pressure. The TUI View Engine LLD must define viewport batching, render scheduling, and latency measurement points.
2. **NFR-PERF-002 (Hot-reload completes within 2 seconds)** — `ADR-008` assigns hot-reload validation and publication to the Configuration Service. LLD must define parser/schema/per-index timing budgets for 10 configured servers.
3. **NFR-REL-007 (Config deletion stability)** — `ADR-008` requires last-known-good snapshot retention. LLD must specify file watcher retry/backoff and operator warning behavior.
4. **NFR-SEC-009 (Read-only Mode)** — `ADR-008` requires central command-router enforcement. LLD must define the read-only allow/deny table and user-facing rejection messages.
5. **NFR-MNT-003 (Validation Tooling `--validate-config`)** — `ADR-008` requires validation-only bootstrap. LLD must define exit codes and validation output format.

There are **no orphan NFRs** in this matrix. All 42 Non-Functional Requirements have been mapped to their respective ADRs, explicit `_Non-Architectural_` records, or approved `_Out of Scope_` decisions.
